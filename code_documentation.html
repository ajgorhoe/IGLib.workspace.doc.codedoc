Intructions for generation of code documentation

  Documentation is not included in SVN repository because it consists of automatically generated files and would load the server unnecessarily. Every user gneerates its own local version of documentation in this working copy of this directory, and the generated files are ignored when SVN commit is performed.

Warning:
If you create a new documentation configuration which will result in a new documentation directory, make sure to set the svn_ignore property on the root directory of code documentation before committing anything to SVN! This will add the newly created directories to the ignore list. Do not by any chance commit directories containing the generated documentation (by convention, automatically generated documentation is located in directories named generated, in order to be distinguishable and can be added to the SVN Ignore list)! Note that when a new directory is created inside a directory that is included in the SVN-ignore list, its recursive contents are not added automatically to the ignore list before you update the list on the directory wheer it is defined.

Important:

If you generate a specific documentation for the first time, you will need to create the directory named "generated"! Otherwise, generation of documentation can fail without a meaningful explanation. Take care that the directory is not committed to SVN (if necessary, add the directory to SVN ignore list).

Doxygen:

  In order to update the documentation (i.e. to generate documentation anew), run the following command from the directory containing the appropriate configuration file:

    doxygen <configuration_file>

where <configuratioon_file> is the path to the configuration file containing all the settings used for generation of documentation. If you want to change some settings, you can edit these files in any text editor. Remember however that these changes wil be committed to SVN repository and will affect all users. Currently, the abiity of having personal setting files is not supported, but this could be added relatively easily.
  There are also shell scripts for Windows provideed for generation of documentation. To generate documentation for a single project, run:

    generate.bat <project_name>

where <project_name> is name of the documentation to be generated (i.e. name of the Doxygen configuration file without extension, e.g. iglib for iglib.dox). Running just the generate.bat script generates documentation for all projects included in the script. Scripts must be run from the directory where they are located. Advantage of generating documentation by running scripts is that scripts may also perform some other operations, e.g. copy some external files to documentation directories.
  There are many settings in doxygen configuration files. In order to make things easier, special marks are put in comments preceeding individual settings.
  The "??" mark means that the effects of the given settings have not been tested or are not clear enough, but the effect might be relatively important.
  The "$$" mark is used to denote the settings that will probably change when copying the configuration file in order to configure a new project. Typically these settings are related to locations of documented files, locations of output and similar.
  There is also a GUI-based tool for editing the settings. Use of this tool is not recommended for changing file and directory paths because using current paths as starting point does not work well. The graphical configuration tool can be run by the following command:
    doxywizard <configuration_file>

If you want to create a new configuration (i.e. you want to create another documentation), the best way is to copy an existing configuration file and change settings that are specific to what is documented (especially those containing file paths). If for some reason you want to start with default settings then you can generate a new configuration file by typing the following command:
    doxygen -g <configuration_file>

Generating Documentation for Python Code:

  In order to generate documentation for Python code wiht Doxygen, you need to use one of the special filters. There are two options:
  • doxypy 
  • doxypypy - this is a newer fork but it does not work for all cases. The previous one is currently more stable.
  Both filters are python scripts/modules which must be installed on the system where documentation is generated (of course, Python itself must be installed on that system). When Doxygen generates documentation, it sends each file to filter scripts that convert the format of python doc comments into a form understandable to Doxygen.

  In order for the filters to work, the following lines must be inserted in Soxygen settings file:
FILTER_SOURCE_FILES = YES
INPUT_FILTER = "python c:\Python27\Scripts\doxypy.py"
 
  The last filter's home has nice instructions on this. A bit more detailed description on how to generate documentation for Python code by Doxygen can be found here:
  • Automatic Documentation of Python Code using Doxygen

SHFB (Sandcastle Help File Builder):

  In order to generate documentation, open the SHFB project file in the graphical application (extension .shfbproj), click on the Documentation menu, and select "Build Project" (there is also a keyboard shortcut, Ctrl-Shift-B).
Warning: Before you start the generation process, all documenteed projects must be compiled with the "XML Documentation" option turned on (in Visual Studio's Solution Explorer, right-click the project name, select "Properties" from the menu, then in the "Build" tab check the checkbox next to "XML Documentation File:").

  In order to change configuration or to create a new one, use the Project Explorer and Project Properties windows. In order to make a new documentation, copy an existent project and rename it (you can also create a new one, but take care that the OutputPath property is appropriately set and points to a directory whose contents can be deleted without harm).

  In the Project Explorer window, under "Documentation Sources" section, you can define which source files, projects and solutions are included in documentation. You can simply drag files there or right-click the the "Documentation sources" and select "Add Documentation Source...".

  Under the "References" section of the Project Explorer window, you can define projects or files that the document code depends on. This information is used in order to show the complete inheritance hierarchy for classes that inherit from classes defined in these projects.

  In the Project Properties window, you can set various settings defining how documentation is generated. The most important is the OutputPath property that defines the directory in which documentation is generated.

Warning: Before generating documentation, make sure that OutputPath property does not point to any directory used for something else but the generated documentation. When documentation is generated, the contents of this directories are deleted!
